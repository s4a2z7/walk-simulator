# Step 1 산출물 — 프로젝트 계획서(목표/팀 역할/핵심 AI 기능)

## 1. 프로젝트 목표와 기대 산출물 정의

### 1.1 프로젝트 목표
- 의료·헬스케어 영역의 사용자 요청을 **대화형 인터페이스**로 처리하는 통합 챗봇을 구현한다.
- 기능은 팀원별 **도메인 모듈로 분리 개발** 후, 최종 단계에서 단일 챗봇 경험으로 통합한다.
- 본 프로젝트는 **의료 진단·처방을 수행하지 않는다.**(정보 제공/업무 처리 중심)

### 1.2 기대 산출물(제출/데모 기준) 확정
| 구분 | 산출물 | 설명 | 완료 기준 |
|---|---|---|---|
| 기획 | 기능 범위·책임 경계 문서 | 모듈별 포함/제외, 인터페이스 개요 | 역할 겹침 없이 경계 명확 |
| 설계 | 예약 플로우/상태 전이 등 핵심 흐름 설계 | 주요 유스케이스와 예외처리 | 1분 데모 시나리오 가능 |
| 데이터 | 데이터 수집 계획표 | 소스/수집 방법/제약/담당 | 합법성·현실성·일정 반영 |
| 개발 | 최소 기능 MVP | 모듈별 핵심 기능 동작 | 통합 라우팅으로 연결 |
| 데모 | 시연 스크립트/영상(선택) | 1분 내 시나리오 2~3개 | 실패/예외 포함 가능 |

## 2. 팀 내 역할 분담(기획/데이터/모델링/데모)

### 2.1 역할 분담 표(템플릿)
| 역할 | 담당자 | 책임(해야 하는 일) | 산출물 |
|---|---|---|---|
| 기획(공통) | 공통 | 범위/정의, 책임 경계, 정책 결정 | 기능 정의서, 플로우 |
| 데이터 | 공통 | 소스 조사, 수집/정제, 스키마 | 데이터 수집 계획표 |
| 모델링/AI | 공통 | AI 적용 지점 정의, 프롬프트/룰 | AI 범위 명세, 테스트 케이스 |
| 백엔드/통합 | 공통 | 라우팅, 공통 API/세션 | 통합 설계, API 명세 |
| 프론트/UX | 공통 | 대화 UI, 입력/선택 UX | 화면/대화 UX 문서 |
| 데모/발표 | 공통 | 시나리오 구성, 리허설 | 데모 스크립트, 발표 자료 |

### 2.2 모듈 단위 책임 경계(예시)
| 모듈 | 담당 | 포함 | 제외 |
|---|---|---|---|
| 예약 추천(병원/약국/검진센터) | 최유림 | 업체/방문유형/준비사항/경로(전화·링크) 추천, 추천 결과 저장/공유 | 실제 예약 확정/변경/취소 대행, 진단/처방, 의료기록 |
| 영양/알러지 관리 | 임소은 | 영양 기록/조회(식단/섭취), 알러지 프로필(알러젠/주의사항) 관리 | 예약 확정/업무 처리, 진단/처방 |
| 활동/걸음 수 | 서유경 | 걸음 수 조회/추이 | 예약/의료 판단 |

## 3. 사용할 AI 기능 범위 확정(포함/제외)

### 3.1 포함(AI 적용)
- **의도 분류/라우팅 보조**: 사용자의 요청을 예약/영양·알러지/활동 모듈 중 어디로 보낼지 판단 보조
- **증상 기반 방문 유형 추천(예약 보조)**: 예약 카테고리 선택을 돕는 수준(진료/검진/약국 수령 등)
- **요약/정규화**: 사용자의 자유 텍스트를 구조화(예: 방문 목적, 메모 요약)

### 3.2 제외(명시적 금지)
- **의료 진단·처방·약 추천**
- 응급도 판정 및 의료기관 선택 강제
- 민감정보(주민번호 등) 과도 수집

### 3.3 공통 출력 원칙(안전/책임 경계)
- AI 결과는 “추천/정리”로만 제공하며, **최종 결정은 사용자 확인(컨펌)**을 거친다.
- 의료적 확정 표현을 피하고, 필요한 경우 일반적 안내 수준으로 제한한다.
