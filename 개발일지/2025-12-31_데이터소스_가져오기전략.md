# 개발일지 — 데이터 소스/가져오기 전략(예약 모듈 중심)

- 작성시각: 2025-12-31 (수)
- 범위: 예약 모듈에서 필요한 운영 데이터(업체/슬롯/정책)를 “어디서/어떻게” 가져올지 설계

---

## 해결하고자 한 문제
- 예약 기능 구현을 위해 필요한 데이터(업체 목록/가용 슬롯/정책)를 **어떤 소스에서** 가져올지 선택해야 한다.
- 소스별 장단점을 비교해 **현실적인 권장안**을 제시해야 한다(팀 프로젝트 일정 고려).
- 데이터 “없음/중복/정합성 불일치” 등 엣지 케이스 처리 원칙을 정해야 한다.

---

## 오늘까지 해결된 것(완료)
- 데이터 소스 옵션(제휴 API/공공 데이터/지도 API/애그리게이터/내부 Seed+모킹)을 비교했다.
- 제출/데모 기준 권장 조합을 **E(내부 Seed+모킹) + (B 또는 C)** 하이브리드로 제시했다.
- `ProviderCatalog / Availability / Booking` 인터페이스로 소스를 추상화하는 구조를 제안했다(향후 교체 가능).
- 데이터 모델 예시(JSON)와 엣지 케이스 처리(데이터 없음/중복/정합성/중복 생성 방지) 규칙을 정리했다.
  - 파일: `데이터소스_가져오기전략.md`

---

## 아직 해결되지 않은 것(미완료/리스크)
- 실제 외부 소스(B/C/D) 선택 여부는 약관/쿼터/프로젝트 제약을 확인 후 팀 합의가 필요하다.
- “가용 슬롯”을 외부에서 가져오지 않는 경우(모킹)에도, 데모에서 납득 가능한 수준의 정책/데이터 구성이 필요하다.

---

## 향후 개발을 위한 컨텍스트(메모)
- 핵심은 “실제 데이터 연동”이 아니라, **예약 업무 처리(상태머신/정책/예외)를 검증 가능한 구조**를 먼저 완성하는 것.
- 외부 데이터 연동은 선택 사항으로 두고, 인터페이스 교체로 단계적 확장 가능하도록 설계한다.
